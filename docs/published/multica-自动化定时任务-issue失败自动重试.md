# 让 Agent 自己“死磕”任务：我用 Multica 的“自动化”定时任务实现 Issue 失败自动重试

> 本文首发于知乎「Agent 工坊」专栏，记录我用 Multica 管理 Agent 工作流的真实过程。

## 一、起因：Agent 跑任务，最怕“中途暴毙”

把活儿交给 AI Agent 去做，最理想的状态是：我提一个需求，写成一个 Issue，剩下的交给 Agent，它自己写代码、跑测试、提 PR，完成后把状态翻到 done。

但现实没这么顺。Agent 在执行过程中会因为各种**非业务原因**挂掉：

- 模型服务端的网络连接被重置（`Connection reset by peer`）；
- 响应流中途断开、超时；
- 上游 API 限流、5xx 抖动；
- 偶发的内部错误。

这些失败和“任务本身做不出来”是两码事——它们是**瞬时故障（transient failure）**。如果一遇到这种错误整个 Issue 就卡死在半路，那我就得时刻盯着面板，手动去重启任务。这就违背了“把活儿交出去”的初衷。

我想要的是：**任务失败了，系统自己重试，一直重试到 Issue 真正进入完成状态为止，中间不需要我介入。**

下面就讲讲我在 Multica 上是怎么把这套自动重试跑通的。

## 二、核心思路：Issue 状态机 + 触发器驱动的重试循环

Multica 的 Issue 不是一张静态卡片，而是一个**状态机**。每个 Issue 都有明确的状态：

```
backlog → todo → in_progress → in_review → done
                      ↓
                   blocked / cancelled
```

关键点在于：**Agent 的运行是由状态变化或评论“触发”的。** 当一个 Issue 被分配给某个 Agent 且进入待处理状态时，平台会给这个 Agent 入队一次运行（run）。Agent 起来后做三件事：

1. 把状态翻到 `in_progress`，宣告“我开始干了”；
2. 执行任务（读 Issue、读评论历史、写代码、跑验证）；
3. 完成后把状态翻到 `in_review`（或 `done`）。

自动重试，就是建立在这个状态机之上的一层**外部守护逻辑**。它要回答三个问题：

- **怎么判定一次 run 失败了？** Agent 异常退出会在 Issue 上留下一条系统（system）评论，内容就是底层错误，比如 `Connection reset by peer`、`Internal error`、`response stream error`。如果一次 run 结束后，Issue 状态没有前进到最终状态，而最近一条记录恰恰是这种系统错误，那就认定失败了。
- **怎么触发下一次重试？** 普通评论不会触发 Agent，只有**带 agent mention 的评论**才会给对应 assignee 重新入队一次 run。所以重试动作就是：在失败的 Issue 下，发一条 `@` 了它 assignee 的评论。
- **什么时候停止？** 进入 `done`/`blocked`/`cancelled` 这类最终状态就停；或者重试次数超过上限（我设的是 10 次）就熔断，改发一条提示需要人工介入的评论。

**最关键的是——我并没有自己写脚本来跑这套逻辑。** 这些判定和重试，全部交给了 Multica 前端「自动化」里的一个**定时任务**去做。下面就照实讲讲我这个定时任务是怎么建的。

## 三、落地：在「自动化」里建一个定时巡检的 Autopilot

Multica 前端有一个「自动化」入口，可以创建 **Autopilot**——也就是“定时 / 触发式”地让某个 Agent 自动跑起来。我做的事情很简单：建一个 Autopilot，让一个“监控员”Agent 每 10 分钟自动巡检一遍 workspace，发现失败就去重启。整个过程没有一行 shell 脚本，全是在页面上点出来 + 用自然语言把规则写清楚。

具体配置如下：

**1. 基本信息**

- **标题**：`multica-monitor`（一眼能看出这是个监控巡检任务）。
- **执行模式**：`run_only`。这点很重要——它表示“每次触发就让指定 Agent 直接跑一轮”，而**不是**每次去新建一个 Issue（那是 `create_issue` 模式）。我要的是周期性地跑一段巡检逻辑，所以选 `run_only`。
- **执行 Agent（assignee）**：指定一个专门的“监控员”Agent 来跑这个巡检任务。它不参与具体业务，只负责扫一圈、发重启评论。

**2. 定时触发器（这就是“定时任务”的核心）**

给这个 Autopilot 加一个 **schedule（定时）触发器**：

- **Cron 表达式**：`*/10 * * * *`，也就是**每 10 分钟**触发一次。
- **时区**：`Asia/Shanghai`（按北京时间来算，省得自己换算 UTC）。

加完之后，平台就会按这个节奏，每 10 分钟自动把“监控员”Agent 拉起来跑一遍，并在详情里显示下一次运行时间（`next_run_at`）。这一步就是字面意义上的“加了一个定时任务”——没有服务器、没有 crontab、没有脚本，纯页面配置。

**3. 把重试规则写进任务描述（用大白话，而不是代码）**

`run_only` 模式下，Autopilot 的**描述（description）就是交给 Agent 的任务 prompt**。所以我不写脚本，而是把“怎么算失败、怎么重启、什么时候停”这些规则用自然语言写清楚，让监控员 Agent 自己照着做。我实际写的规则大致是这几块：

- **目标**：每 10 分钟检查一次 workspace 里**尚未进入最终状态**的 issue，发现 agent 任务明确失败、但 issue 还没到最终状态的，就自动重启它的 assignee。

- **哪些状态算“最终状态”**：`done`、`blocked`、`cancelled` 绝不重启；`todo`、`backlog`、`in_progress`、`in_review` 以及各种 working/running 中间态，才是“还没完成、可以重试”的对象。

- **怎么判失败**：只处理**明确失败**的——最近一条 agent run / 系统评论里出现 `failed`、`error`、`Internal error`、`response stream error`、`connection reset`、运行时/工具失败之类的字样，且之后没有成功完成的记录。对那些还在正常跑、或刚被触发还在排队的，不动它，避免重复启动。

- **怎么重启（关键）**：不要只评论“继续”——普通评论不触发 Agent。要先读 issue 详情，确认 `assignee_type` 是 agent 且有 `assignee_id`，然后在该 issue 下发一条**带 agent mention** 的评论，格式是：

  > 请继续处理这个任务，上一轮执行失败但 issue 尚未进入最终状态。`[@AGENT_NAME](mention://agent/AGENT_ID)`

  正是这个 `mention://agent/...` 链接，才会给对应 Agent 真正入队一次新的 run。

- **重试上限（熔断）**：每个 issue 最多自动重启 10 次（靠统计本自动化此前在该 issue 下发过的重启评论数量来判断）。到 10 次还没成功，就不再 mention，改发一条“已自动重启 10 次仍未成功，停止自动重试，需要人工检查”的评论，把球踢回给我。

- **去重与安全**：没有 agent assignee 的 issue 不强行 mention，只记录跳过；已经有新 run 在跑的不重启；同一轮巡检每个 issue 最多触发一次；如果上一条重启评论之后并没有新的失败记录，就不重复重启——这些规则就是为了防止“重复入队跑两遍”和无意义互相触发。

把这几块写进描述，保存。从此这个 Autopilot 就成了一个全自动的“守夜人”。

## 四、为什么这套“定时任务 + 自然语言规则”能 work

它能跑通，靠的是 Multica 的两个机制叠加：

1. **定时触发器**负责“每隔 10 分钟巡一遍”的节奏——这是平台帮我做的，我只填了个 cron。
2. **mention 触发**负责“真正把失败 Agent 重新拉起来”——监控员 Agent 发一条 `@` 评论，平台就给目标 Agent 入队一次 run。

而被重启的 Agent 每次起来都会**强制重读完整的评论历史**，所以它能看到上一次失败留下的报错、之前建过的分支、提过的 PR。重试不是“无脑从头再来”，而是**带着上下文续跑**——这才是“失败自动重试”有意义的核心。

## 五、几个经验

1. **区分“瞬时故障”和“任务做不出来”。** 网络重置、流断开该重试；但如果是需求本身有歧义、Agent 反复卡在同一步，重试只会原地打转。所以规则里特意写了“只处理明确失败”和“10 次熔断”，到点就转人工。

2. **一定要有熔断上限。** 没有上限的自动重试，遇到死任务就是无底洞。靠“统计已发的重启评论数”来卡 10 次，简单可靠。

3. **重启必须用 mention，不能只评论“继续”。** 这是我一开始踩的坑——普通评论根本不会触发 Agent。只有 `mention://agent/<id>` 才会真正入队一次 run。

4. **用 `run_only` 而不是 `create_issue`。** 我要的是“周期性跑一段巡检逻辑”，不是“周期性建新 issue”，所以模式选对很关键。

5. **结果都落在评论里。** 无论重启还是熔断，都写成 issue 评论——既是给我看的交付物，也正好成了下一次重试的上下文来源，一举两得。

## 六、小结

我用 Multica 实现“Issue 失败自动重试直到完成”，全程**没写一行脚本**，只在前端「自动化」里建了一个 Autopilot：

- **`run_only` 模式 + 一个监控员 Agent**：负责执行巡检；
- **`*/10 * * * *` 的定时触发器（Asia/Shanghai）**：每 10 分钟自动跑一次；
- **一段自然语言的任务描述**：把“怎么判失败、用 mention 重启、10 次熔断、去重安全”这些规则讲清楚。

剩下的就交给它了——定时巡检、发现失败、带上下文重启、超限熔断转人工。我把需求写成 Issue 之后，这套“定时任务”就替我盯着，真卡住了才发评论叫我。
