# 一文讲透 PyPI 包的版本号：从 PEP 440 到实战选型

每次 `pip install` 时，pip 都要在一堆版本里挑出一个装给你。它凭什么认为 `1.10` 比 `1.9` 新？为什么 `1.0rc1` 默认装不上、而 `1.0` 可以？`2.1.0+cu118` 这种带加号的版本又是干嘛的？这些问题的答案，全都写在一个叫 **PEP 440** 的规范里。

这篇文章把 PyPI（准确说是 Python 打包生态）的版本号规则梳理清楚：先讲版本号的整体结构，再讲公共版本号的排序规则，最后讲“最终版 / 预发布版 / 发布后版 / 开发版”各自的适用场景。读完你应该能自己设计一套合理的版本号方案，也能看懂任意一个 PyPI 上的版本字符串到底是什么意思。

## 一、版本号的整体结构：公共版本 + 本地版本

PEP 440 规定，一个完整的版本标识符由两部分组成，用 `+` 分隔：

```
<public version>[+<local version>]
   公共版本号        本地版本号（可选）
```

- **公共版本号（public version）**：`+` 左边的部分，是发布到公共索引（PyPI）时唯一允许使用的版本号。它有严格的结构和排序规则，是本文的主角。
- **本地版本号（local version）**：`+` 右边的部分，可选。用来标记“基于某个公共版本、经过本地/下游修改”的构建。一般只在私有源、内网、本地或特定分发渠道流通时，才使用 local 版本标识符。

举几个例子：

| 完整版本 | 公共版本 | 本地版本 |
| --- | --- | --- |
| `1.4.2` | `1.4.2` | 无 |
| `1.4.2+cpu` | `1.4.2` | `cpu` |
| `2.31.0+companyfix.1` | `2.31.0` | `companyfix.1` |

**本地版本号的关键规则**：

- 格式为 `[a-z0-9]+(\.[a-z0-9]+)*`，由字母数字和点组成；下划线、连字符在规范化时会被统一成点。
- **公共 PyPI 明确禁止上传带本地版本的包**。它的设计意图就是“公共索引之外”的标识。
- 它表示“公共版本语义不变，但这是某个特定的重打/重编构建”。

**典型使用场景**：

1. **打了补丁的重新构建**：拿上游 `requests 2.31.0` 打了安全补丁、API 没变，重新打包成 `2.31.0+companyfix.1`，既表明还是 2.31.0 那套接口，又能和官方原版区分。
2. **同一版本的不同硬件/编译变体**：最经典的是 PyTorch —— `2.1.0+cpu`、`2.1.0+cu118`、`2.1.0+cu121` 是同一个 `2.1.0`，分别对应 CPU、CUDA 11.8、CUDA 12.1 的编译产物。
3. **内部/私有重打包标识**：内部 CI 重打外部依赖时贴来源标记，如 `1.4.2+internal.20240130`，方便排查机器上到底装的是不是自家重打的包。
4. **本地开发临时构建**：本地 `pip install .` 时打个 `+dev.<git短哈希>`，区别于正式发布物。

一句话总结：**本地版本号是给“下游二次分发者”用的，不是给“上游发布者”用的。** 你往公共源发包时几乎用不到它。

## 二、公共版本号的完整结构

公共版本号本身又可以拆成若干段，从左到右是：

```
[N!]N(.N)*[{a|b|rc}N][.postN][.devN]
 epoch  release   pre-release   post     dev
 纪元    发布段     预发布段      发布后段  开发段
```

- **Epoch（纪元，`N!`）**：默认 `0`，通常省略。比较优先级最高，是“推倒重来”的逃生舱。
- **Release 段（`1.2.3`）**：核心版本号，最常见的部分。
- **Pre-release 段（`{a|b|rc}N`）**：预发布标记，可选。
- **Post 段（`.postN`）**：发布后修订，可选。
- **Dev 段（`.devN`）**：开发版标记，可选。

后面四类（pre / post / dev 加上纯 release）就对应了本文要重点讲的四种“版本类型”。先把排序规则讲清楚，再讲它们各自的适用场景。

## 三、公共版本号的排序规则

pip 选版本的本质就是：**在所有合规候选里，按 PEP 440 排序选最大的那个。** 所以理解排序规则至关重要。比较是**逐段、从左到右**进行的。

### 1. Epoch（纪元）

最高优先级，默认 `0`。`1!1.0` 比任何 `0!x.y`（也就是普通的 `2.0`、`99.0`）都大。

它的用途：当你彻底改了版本号方案、新方案在数值上反而比旧的小，用 epoch 强行抬高。比如某项目从日期版本 `2023.1` 切换到语义版本 `1.0`，直接发 `1.0` 会被认为比 `2023.1` 旧，于是发 `1!1.0`。日常极少见。

### 2. Release 段

按数字逐段比较：`1.2.0 < 1.2.1 < 1.3.0 < 2.0.0`。

两个关键点：

- **按整数比，不是字符串**：`1.10 > 1.9`（十大于九），而不是按字符串 `"1.10" < "1.9"`。
- **长度不同时缺位补 0**：`1.2 == 1.2.0`，`1.2 < 1.2.1`。

Release 段可以有任意多级，但实践中常用三级（`主版本.次版本.修订`，即语义化版本 SemVer 的形式）。

### 3. Pre-release 段（最反直觉的一点）

预发布版排在**同号正式版之前（更小）**。三种类型按从早到晚 / 从小到大排序：

```
1.0a1 < 1.0a2 < 1.0b1 < 1.0rc1 < 1.0
```

- `a` = alpha（内测）
- `b` = beta（公测）
- `rc` = release candidate（候选发布）

后面的数字是同类型内的递增序号，省略按 `0` 算（`1.0a` == `1.0a0`）。

**规范化（normalization）是大坑**：PEP 440 允许多种写法，但工具会统一成标准形式，下面这些是等价的：

- `1.0alpha1` → `1.0a1`，`1.0beta1` → `1.0b1`
- `1.0c1` / `1.0pre1` / `1.0preview1` → `1.0rc1`（`c` 和 `rc` 等价）
- 分隔符 `-`、`_`、`.` 和大小写都会被吸收：`1.0-A.1`、`1.0_a1`、`1.0.A1` 全部归一为 `1.0a1`

所以你写 `1.0.0-rc.1`、`1.0.0RC1`、`1.0.0rc1`，最终在 PyPI 上都是同一个版本 `1.0.0rc1`。

### 4. Post 段

排在同号正式版**之后（更大）**：`1.0 < 1.0.post1 < 1.0.post2`。

### 5. Dev 段

优先级**最低**，把所属版本往前拉：`1.0.dev1 < 1.0a1`，甚至 `1.0a1.dev1 < 1.0a1`。

### 6. Local 段

带 local 的比同号不带的**大**：`1.0 < 1.0+local`。local 段内按点分段比较，数字段按数值、字母段按字典序，且数字段 > 字母段。

### 排序直觉链

把同一个 `1.0` 的各种形态从小到大排一遍：

```
1.0.dev1  <  1.0a1.dev1  <  1.0a1  <  1.0b1  <  1.0rc1  <  1.0  <  1.0.post1  <  1.0+local
```

记住三条核心直觉就够了：

- **dev 永远最早**（把版本往前拉）；
- **pre-release（a/b/rc）排在正式版之前**；
- **post 排在正式版之后**。

可以用标准库随时验证：

```python
from packaging.version import Version
print(Version("1.0rc1") < Version("1.0"))   # True
print(sorted([Version(v) for v in ["1.0", "1.0rc1", "1.0.dev1", "1.0.post1"]]))
# [<Version('1.0.dev1')>, <Version('1.0rc1')>, <Version('1.0')>, <Version('1.0.post1')>]
```

## 四、四种版本类型的适用场景

这是最实用的一节。PEP 440 把版本分成四类，每一类都有明确的语义和使用时机。

### 1. 最终版本（Final release）

形如 `1.0`、`2.3.1` 的纯 release 版本号，没有任何后缀。

- **含义**：正式、稳定、面向所有用户的发布。
- **适用场景**：这是绝大多数发布应该用的形式。功能完成、测试通过、可以交付给普通用户使用。
- **pip 行为**：`pip install pkg` 默认就是从最终版本里选最高的。这是最“正常”的版本类型。

实践建议：最终版本号通常遵循语义化版本（SemVer）`主版本.次版本.修订`：

- 主版本（major）：不兼容的 API 变更；
- 次版本（minor）：向后兼容的新功能；
- 修订号（patch）：向后兼容的 bug 修复。

### 2. 预发布版本（Pre-release）

形如 `1.0a1`、`2.3.0b2`、`1.5rc1`，带 `a`/`b`/`rc` 后缀。

- **含义**：发布前的测试版本，让早期用户提前试用、收集反馈。
- **三个阶段的语义**：
  - **alpha（a）**：内测，功能可能不全、不稳定，主要给开发团队和极早期用户；
  - **beta（b）**：公测，功能基本齐全，开放给更大范围用户测试；
  - **rc（release candidate）**：候选发布，已经认为可以发正式版了，做最后验证，没问题就直接发同号正式版。
- **适用场景**：大版本发布前的灰度测试、让用户提前适配新 API、CI/CD 中的预发布通道。
- **pip 行为（关键）**：`pip install pkg` **默认跳过预发布版**，普通用户感知不到它。想装到必须显式声明：
  - `pip install --pre pkg`（允许预发布，取最高的预发布版）；
  - `pip install pkg==1.0rc1`（精确点名，即使不加 `--pre` 也能装）；
  - 约束本身只能匹配到预发布版时（如 `pkg>=1.0rc1`），pip 也会放行。

这个“默认跳过”的特性非常有用：你可以把预发布版和正式版发到**同一个源**，普通用户的 `pip install pkg` 照旧拿最新正式版，只有知道“暗号”（`--pre` 或精确版本）的用户才装得到预发布版。

### 3. 发布后版本（Post-release）

形如 `1.0.post1`、`2.3.1.post2`，带 `.postN` 后缀。

- **含义**：在某个正式版**发布之后**做的微小修订，但这些修订**不涉及代码本身的实质功能变更**。
- **适用场景**：
  - 只修正了打包元数据、README、文档，代码一字未改；
  - 修正了上传时的错误（比如漏打了某个文件），需要重新发布但又不想占用一个新的修订号；
  - 强调“这还是 1.0，只是发布层面的小修补”。
- **pip 行为**：post 版本会被当作正常版本参与选择（不像 pre 那样被跳过），且排在同号正式版之后。所以 `pip install pkg` 会**优先选到 `1.0.post1` 而非 `1.0`**。
- **注意**：post 版本不适合用来发布代码 bug 修复——那种情况应该用新的修订号（`1.0.1`）。post 仅用于“非代码”的发布层面修正。

### 4. 开发版本（Development release）

形如 `1.0.dev1`、`1.0a1.dev2`，带 `.devN` 后缀。

- **含义**：最“早”的、未完成的开发快照，优先级低于一切。
- **适用场景**：
  - **CI/CD 自动构建**：每次提交自动打一个 `1.1.0.dev{commit计数}` 或 `1.1.0.dev{时间戳}`，发到内部开发源，供持续集成、夜间构建（nightly build）使用；
  - **持续部署的中间产物**：还没到 alpha 阶段、纯内部流转的构建；
  - 标记“正在朝 1.1.0 开发，但还远未到可测试阶段”。
- **pip 行为**：和预发布版一样，`pip install pkg` **默认跳过 dev 版本**，需要 `--pre` 或精确指定才能装到。

### 四类对比速查

| 类型 | 示例 | 相对正式版位置 | pip 默认是否可见 | 主要场景 |
| --- | --- | --- | --- | --- |
| 开发版 dev | `1.0.dev1` | 之前（最早） | 否 | CI 自动构建、nightly |
| 预发布 pre | `1.0a1`/`1.0b1`/`1.0rc1` | 之前 | 否 | 灰度测试、提前试用 |
| 最终版 final | `1.0` | —— | 是 | 正式交付（默认选这个） |
| 发布后 post | `1.0.post1` | 之后 | 是 | 打包/文档等非代码修正 |

## 五、把它们串起来：一次典型的版本演进

以一个项目从开发到发布 `1.2.0` 的全过程为例，版本号大致会这样演进：

```
1.2.0.dev1, 1.2.0.dev2, ...   ← CI 每次提交自动打，纯内部
1.2.0a1, 1.2.0a2              ← 内测，给早期用户
1.2.0b1                       ← 公测，扩大测试范围
1.2.0rc1, 1.2.0rc2           ← 候选发布，最后验证
1.2.0                        ← 正式发布！普通用户 pip install 拿到的就是它
1.2.0.post1                  ← 发现 README 写错了，重新发个非代码修正
1.2.1                        ← 发现代码 bug，发修订版（不是 post！）
```

这条链清晰地展示了四种类型各自的位置：dev 在最前、pre（a→b→rc）紧随其后、final 是分水岭、post 在其后做非代码修补，而真正的代码修复则进入下一个 release 号。

## 六、实践建议小结

1. **正式发布永远用最终版本号**（`主.次.修订`），遵循语义化版本。
2. **要做发布前测试就用预发布版**（`a`/`b`/`rc`），利用 pip 默认跳过的特性，可与正式版共用一个源做灰度。
3. **post 版本只用于非代码的发布层面修正**，代码 bug 修复请用新的修订号。
4. **dev 版本交给 CI 自动化**，用于内部构建和 nightly。
5. **本地版本号（`+xxx`）只在私有/内部渠道使用**，公共 PyPI 禁止上传。
6. **不要自创版本格式**（如 `1.0-test`、`1.0.beta`），不符合 PEP 440 的字符串要么被规范化成你没预期的样子，要么直接被工具拒绝。拿不准时用 `packaging.version.Version` 打印出规范化后的真实版本号。

把版本号设计好，本质上是在向使用方传递清晰的“稳定性承诺”。理解了 PEP 440 的这套分层规则，你就能让 pip 的默认行为自动帮你把不同成熟度的版本分发给不同的人群——这正是 Python 打包生态最优雅的设计之一。
